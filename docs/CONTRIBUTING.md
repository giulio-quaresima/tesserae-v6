# Contributing to Tesserae V6

## Overview

Tesserae V6 is a Flask + React web application for detecting textual parallels across a corpus of 2,100+ classical literary works. This document describes how to contribute code.

## Getting Started

1. **Clone the repository:**
   ```
   git clone https://github.com/tesserae/tesserae-v6.git
   cd tesserae-v6
   ```

2. **Create a branch from `main`:**
   ```
   git checkout main
   git pull origin main
   git checkout -b your-name/feature-description
   ```
   Use a descriptive branch name, e.g. `jane/add-timeline-visualization` or `alex/fix-greek-lemmatization`.

3. **Set up your environment:**
   - Python 3.12+, Node.js 18+
   - `pip install -r requirements.txt`
   - `cd client && npm install`
   - Copy `.env.example` to `.env` and fill in database credentials (ask Neil)

## Development Workflow

1. Make your changes on your branch. Commit frequently with clear messages.

2. **Keep your branch up to date with `main`.** Pull from `main` regularly (at least every few days) to avoid large merge conflicts later:
   ```
   git checkout main
   git pull origin main
   git checkout your-branch
   git merge main
   ```
   If there are conflicts, resolve them and commit. Small, frequent merges are much easier than one big merge at the end.

3. **Run the test suite before submitting:**
   ```
   python tests/test_endpoints.py
   ```
   All tests must pass. If you've added new functionality, consider adding tests for it.

4. **Build the frontend if you changed any client/ files:**
   ```
   cd client && npm run build
   ```

## Submitting Your Work

1. **Push your branch to GitHub:**
   ```
   git push origin your-name/feature-description
   ```

2. **Open a Pull Request (PR) to `main`** on GitHub. In the PR description:
   - Summarize what you changed and why
   - Note any new dependencies or environment changes
   - Mention which features/pages are affected

3. **What happens next:**
   - Neil will pull your branch to the dev server for testing
   - The endpoint test suite (29 tests) will be run against your changes
   - Neil may leave review comments on the PR — address them and push again
   - Once everything passes, Neil merges the PR into `main`
   - Changes are then deployed to production

## Code Organization

| Directory | What's There |
|-----------|-------------|
| `backend/` | Flask application, API routes, search logic |
| `backend/blueprints/` | API endpoint modules (search, hapax, fusion, etc.) |
| `client/src/` | React frontend source |
| `client/src/components/` | React components organized by feature |
| `dist/` | Built frontend (generated by `npm run build`) |
| `tests/` | Endpoint tests |
| `texts/` | Corpus `.tess` files (not in git — too large) |
| `data/` | Search indexes, lemma tables (not in git) |

## Important Technical Notes

### API Prefix

Routes are registered **without** the `/api` prefix in blueprint files and with `@api_route(...)` in `app.py`. The `/api` prefix is added at registration time based on the environment:

- **Production (behind Apache):** Apache's `WSGIScriptAlias /api` strips the prefix before Flask sees requests. `API_PREFIX = ""`
- **Dev server (`python main.py`):** Flask handles the full URL. `API_PREFIX = "/api"`

**Do not hardcode `/api` in blueprint route decorators.** Use `@your_bp.route('/your-endpoint')` and the prefix will be handled automatically.

### Blueprint Pattern

If you're adding a new blueprint:

```python
from flask import Blueprint
my_bp = Blueprint('my_feature', __name__)

@my_bp.route('/my-endpoint', methods=['GET'])
def my_endpoint():
    ...
```

Register it in `backend/app.py`:
```python
app.register_blueprint(my_bp, url_prefix=API_PREFIX or None)
```

### Frontend

- React 18 + Vite 5 + Tailwind CSS 3
- Color scheme: red (`bg-red-700`) for primary actions, amber (`bg-amber-600`) for secondary. No green.
- Always rebuild after changes: `cd client && npm run build`
- API calls use `API_BASE = '/api'` (see `client/src/utils/api.js`)

### Data Files

Large data files (texts, embeddings, indexes) are **not in git**. They live on the server filesystem. If your feature needs new data files, document what's needed and where it should go.
